Strategic Thinking

Three Decisions That Defined the Work

The decisions that separated a functional implementation from one that fundamentally changed how the organization operates.

01
Rate Model Before Everything Else
Before a single project was created, I built the full rate model from scratch — accounting for productive hours, travel and expenses, and every cost factor that contributes to the true fully-loaded cost of R&D work. Most implementations skip this step and end up with financial data that looks complete but can’t be reconciled against anything real. Building it right from the start is what allows the platform to tie directly back to the P&L — so when leadership pulls a portfolio report, it reconciles with the numbers their finance team is already working from.
02
Rebuild the Data Model, Not Just the Reports
Exporting data and layering reports on top means losing the table relationships that make the underlying data meaningful. Instead, I rebuilt the full enterprise data model in Microsoft Fabric — starting with an initial full API load to establish the baseline, then automated delta pulls that only retrieve updated fields on each subsequent run. This keeps the Lakehouse continuously current without hammering the source system with repeated large requests. Once curated, the data sits in a governed table that anyone in the organization can query directly, regardless of whether they hold a platform license.
03
Designing an Onboarding Program With No Blueprint
Documentation tells people what a platform does — it doesn’t teach them why it matters to their specific role. With no prior rollout to reference, I designed the training program by role rather than by feature, building each session around the exact workflows that group of users would actually touch day to day. The result was 95% adoption across 1,000+ users on a platform the organization had never used before.

Situation
The right people still could not see the data

The platform was live and generating reliable data — but a large portion of the organization had no licenses, including finance and senior executives. They were still manually reconciling portfolio data from multiple sources, a process that took up to 2 days and carried consistent risk of error. Funding decisions were being made on information that was already outdated the moment it was compiled.

Task
Turn a reporting request into a governed architecture

When leadership asked for better reporting access, I proposed something bigger: rebuilding the full enterprise data model in Microsoft Fabric rather than layering exports on top of flat files. The goal was a governed, always-current source of truth that anyone in the organization could access — no platform license required.

Action
Initial load, delta pulls, governed access

I architected the Lakehouse starting with a full API load to establish the baseline, followed by automated delta pulls that only retrieve updated fields on each run — preventing large repeated requests from impacting the source system. Once curated, the data became accessible org-wide. I then built 10+ Power BI dashboards connecting directly to the Lakehouse, with no manual refresh required.
